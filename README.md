# Дашборд заказов RetailCRM

Мини-дашборд для отображения заказов из RetailCRM. Заказы синхронизируются в Supabase, дашборд отображает их в виде графика и таблицы, а Telegram-бот присылает уведомления о крупных заказах (>50 000 ₸).

## Стек технологий

| Компонент | Технология |
|-----------|-----------|
| Источник данных | RetailCRM API v5 |
| База данных | Supabase (PostgreSQL) |
| Хостинг дашборда | Vercel |
| Уведомления | Telegram Bot API |
| Runtime | Node.js |

## Структура проекта

## С какими проблемами столкнулся

### 1. POST /api/v5/orders возвращал 405

RetailCRM не принимает создание заказов через этот эндпоинт, он только для чтения.

Решение: переключился на POST /api/v5/orders/upload с параметром site и загрузкой пачками по 50 заказов.

---

### 2. HTTP 460 "Orders are loaded with errors"

После перехода на правильный эндпоинт заказы всё равно не грузились. API возвращал failedOrders но без деталей что именно не так.

Решение: отправил один минимальный тестовый заказ без лишних полей. Выяснилось что поле orderType: "eshop-individual" не существует в этом аккаунте. Убрал его через деструктуризацию и все 50 заказов загрузились нормально.

---

### 3. График на дашборде был пустым

Все заказы падали в одну точку на графике потому что в mock_orders.json не было поля createdAt и RetailCRM ставил всем заказам текущую дату.

Решение: разбросал даты в mock_orders.json равномерно за последние 30 дней, написал скрипт fix_dates.js который обновил поле created_at в Supabase без повторной загрузки в CRM.

---

### 4. Телеграм-бот не мог достучаться до api.telegram.org

Node.js падал с ETIMEDOUT, хотя в браузере телеграм открывался нормально. Провайдер блокировал трафик к серверам Telegram на уровне сети.

Решение: включил VPN и бот сразу заработал.
